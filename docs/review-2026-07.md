# Code Review вЂ” July 2026

Review of the Worktide system: backend (Symfony/PHP), SPA (React/Refine), and portal (separate API surface + separate frontend repo).

## Architecture Overview

### Backend (`worktide`)
- **Stack**: Symfony 8.1, API Platform 4.3, Doctrine 3.6, MySQL 8.0, PHP 8.4
- **Auth**: JWT (LexikJWTAuthenticationBundle) + httpOnly refresh cookie + Personal Access Tokens (`wt_pat_*`)
- **Real-time**: Mercure (SSE, not WebSockets)
- **Async**: Symfony Messenger (Doctrine transport, three queues: `async`, `ai_agents`, `search`)
- **Tenant isolation**: Doctrine query extension (`WorkspaceScopeExtension`) + 18 Symfony Voters + `PermissionResolver` (capability-based, not role-based)
- **SSRF guard**: `EgressGuard` (default-deny) + `OutboundUrlGuard` (no `file://`, no internal IPs)
- **AI**: Anthropic Claude + Ollama (local) + Infomaniak (Swiss/EU), per-task-type routing, cost tracking + budgets
- **Tests**: 128 PHP test files, full PHPUnit suite runs in CI with MySQL 8

### SPA (`worktide-web`)
- **Stack**: React 19, TypeScript 6, Vite 8, Tailwind CSS v4, Refine 5
- **Auth**: JWT in memory only (XSS-resilient), httpOnly refresh cookie, single-flight refresh
- **Real-time**: Mercure via `event-source-plus` (fetch-based SSE, supports auth headers)
- **API client**: Custom axios instance with interceptors, offline queue (`pendingQueue`), network classification
- **Codegen**: Kubb-generated types (434 files) and Zod schemas from OpenAPI 3.1 spec
- **Tests**: Vitest 4 вЂ” 2 test files (12 tests) for api + dataProvider. No component tests.
- **CI**: Deployment-only (Coolify webhook). **No CI workflow for tsc/vitest/build.**
- **State**: Refine's data provider + custom hooks (`useKeysetList`, `useUserDirectory`)

### Portal (separate API surface + separate frontend repo)
- **API**: 25 hand-written Symfony controllers under `/v1/portal/*` вЂ” no API Platform
- **Auth**: Same JWT mechanism but `ROLE_PORTAL` only (no `ROLE_USER`)
- **Firewall split**: `^/v1/portal в†’ ROLE_PORTAL` matched before `^/v1 в†’ ROLE_USER`
- **Tenant isolation**: `PortalAccessResolver` resolves identity chain `User в†’ Contact в†’ Customer в†’ allowed projects`
- **Feature gating**: 15 feature keys per workspace (dashboard, tickets, files, monitoring, etc.)
- **User checker**: `PortalUserChecker` re-validates portal access on every request (fail-closed)
- **Frontend**: Separate React SPA repo (`worktide-portal`), not in this codebase

## Findings вЂ” Backend

### Strengths
- вњ… Defence-in-depth tenant isolation (Doctrine extension + Voters + PermissionResolver вЂ” three independent layers)

- вњ… UUIDv7 primary keys (time-ordered, no sequence management)
- вњ… 25 functional API tests guard serialization leaks + tenant isolation

- вњ… CI runs full PHPUnit suite with MySQL 8 on every push + PR

### Risks

- вљ пёЏ **Soft-delete unique constraint collisions**: Re-creating a soft-deleted entity within the 30-day retention window produces a 409. Documented as acceptable.
- вљ пёЏ **IMAP polling**: 2-minute interval вЂ” no IMAP IDLE push. Adequate for most but not real-time.
- вљ пёЏ **Messenger transport**: Doctrine-based by default. Under load, Redis/Valkey switch is needed.
- вљ пёЏ **Repository proliferation**: ~100 repository classes, many are `EntityManager` pass-throughs.

## Findings вЂ” Portal

### Strengths
- вњ… Hard role split: `ROLE_PORTAL` without `ROLE_USER`. Firewall catch-all `^/v1 в†’ ROLE_USER` automatically excludes portal users from staff endpoints.
- вњ… Four security layers: Firewall в†’ `PortalUserChecker` (re-validates every request) в†’ Feature gating (15 keys) в†’ Tenant scoping via `PortalAccessResolver`
- вњ… Fail-closed tenant isolation: `PortalAccessResolver::contact()` returns 403 if Contact was unlinked. `findTicketOr404()` returns 404 (not 403) to avoid revealing existence.
- вњ… Dedicated API surface: Hand-written controllers return curated DTOs, never raw entities. No `WorkspaceScopeExtension` incompatibility.
- вњ… Scoped Mercure token: Portal users subscribe to `{ws}/portal/notifications/{userId}` only, never `*`.
- вњ… White-labelling: `BRAND_*` env vars, no build/deploy for branding changes.
- вњ… Audit trail: `portal.impersonation.issued` / `portal.impersonation.consumed` logged via `DomainEventLog`.

### Risks
- вљ пёЏ **25 controllers parallel to API Platform**: Every portal endpoint manually maintained. New features need double work (API Platform resource + portal controller).
- вљ пёЏ **`isHiddenForConnectUsers` is convention, not invariant**: Docblock in `findTicketOr404` explicitly warns that enforcement is manual for Tasks (unlike Comment/File).
- вљ пёЏ **SLA time-pause simplified**: Pauses only at current `isWaitingForCustomer` status. Historical waiting periods not subtracted.
- вљ пёЏ **Workspace settings untyped**: `Workspace::getSettings(): ?array` вЂ” portal config navigated by string keys with `??`.
- вљ пёЏ **No portal Voters**: Authorization checked imperatively in controllers via `PortalAccessResolver`, not through Symfony's Voter system.
- вљ пёЏ **Password reset gap**: Portal users get a set-password link on initial grant but no dedicated "forgot password" flow.
- вљ пёЏ **Frontend in external repo**: Backend API changes for portal must coordinate across repo boundaries.
